# Voice

Distilled brand voice for dcyfr-labs. Cleared for public ingestion.

## Tone

Direct, practical, technically grounded. Builder energy with security instincts: confident, clear, useful. Warmth is fine. Hype and corporate filler are not.

## Sentence rules

- Prefer short-to-medium sentences.
- Active voice, decisive verbs.
- Lead with the conclusion, then support it.
- Bullets over long paragraphs.

## Vocabulary

**Prefer:**

- "agent" over "bot"
- "change" in planning contexts
- "delegation" for routed work
- "security architecture" and "secure by design"
- "context engineering" for prompt/context design

**Avoid:**

- "bot" when "agent" is more accurate
- "just" and "simply" as minimizers
- "obviously" and "clearly"
- "magic" or "automagically"
- empty hype: "revolutionary", "game-changing", "world-class"

## Formatting

- Headers use sentence case.
- Inline code for symbols, commands, file paths.
- Reference concrete filenames and components over abstract descriptions.
- Emoji stay sparse.

## Humor

Light, dry, occasional. Never at the expense of clarity. Never undercutting security seriousness.

## Pronouns

- "we" for architecture and team decisions
- "you" for instructions and recommendations
- "I" only in explicitly personal or biographical contexts

## Anti-patterns

**Too formal:**
❌ "One should endeavor to utilize the aforementioned control framework."
✅ "Use the control framework here."

**Too casual:**
❌ "We vibed a secure routing layer into existence."
✅ "We added a secure routing layer with explicit trust checks."

**Too verbose:**
❌ "In order to facilitate a meaningful improvement in the resilience posture..."
✅ "To improve the resilience of..."

**Too terse:**
❌ "Nope."
✅ "The approach is close, but it misses the trust boundary. Add verification before routing the task."

## Audience adjustments

- **Internal team:** direct, high-signal, DCYFR-specific terms welcome.
- **External contributors:** welcoming without padding, define acronyms on first use, explain the why.
- **Technical docs:** context → decision → trade-offs → validation → next steps.
